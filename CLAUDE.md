# terraform-azure-postgresql-server

Simple Terraform building block wrapper around Azure PostgreSQL Flexible Server and related resources (`azurerm_postgresql_flexible_server`, `azurerm_postgresql_flexible_server_database`).

## Tech Stack

- **Language**: HCL (Terraform)
- **Provider**: `hashicorp/azurerm` >= 4.20.0
- **Terraform**: >= 1.11
- **Docs generator**: `terraform-docs` with `.terraform-docs.yaml`

## Architecture

Single-layer building block — wraps `azurerm_postgresql_flexible_server` as the primary resource (`"this"`) and `azurerm_postgresql_flexible_server_database` as a secondary resource managed via `for_each`. The module fetches the resource group via a data source (does not create it). No VNet or private-endpoint logic lives here — those are separate building blocks.

### Module structure

```
terraform-azure-postgresql-server/
├── main.tf              # Primary resources: azurerm_postgresql_flexible_server + databases
├── variables.tf         # Input variable declarations
├── outputs.tf           # Output value declarations
├── versions.tf          # Terraform & provider version constraints
├── metadata.tf          # ⚠ DO NOT MODIFY — custom timeout/tag/validator constructs
├── locals.tf            # Module-level locals (default tags, definitions map)
├── resource-group.tf    # data "azurerm_resource_group" "this"
├── README.md            # Auto-generated by terraform-docs
├── .terraform-docs.yaml # Docs generator config
├── Makefile             # format, docs, validate, plan, apply
└── examples/
    └── base/
        └── main.tf      # Minimal usage example
```

## Conventions

### Resource naming
- Main resource is always named `"this"`: `resource "azurerm_postgresql_flexible_server" "this"`
- Secondary resources (databases) use `for_each` with a meaningful key

### Data sources
- Resource group is fetched via `data "azurerm_resource_group" "this"` in `resource-group.tf`
- Never create the resource group inside the module — building blocks consume, not create, shared infrastructure

### Timeouts
All resources must use the `metadata.tf` timeout pattern:
```hcl
timeouts {
  create = try(local.metadata.resource_timeouts["azurerm_postgresql_flexible_server"]["create"], local.metadata.resource_timeouts["default"]["create"])
  read   = try(local.metadata.resource_timeouts["azurerm_postgresql_flexible_server"]["read"],   local.metadata.resource_timeouts["default"]["read"])
  update = try(local.metadata.resource_timeouts["azurerm_postgresql_flexible_server"]["update"], local.metadata.resource_timeouts["default"]["update"])
  delete = try(local.metadata.resource_timeouts["azurerm_postgresql_flexible_server"]["delete"], local.metadata.resource_timeouts["default"]["delete"])
}
```

Only include timeout actions that the provider supports. `azurerm_postgresql_flexible_server_database` does **not** support `update` — only `create`, `read`, `delete`. Only add a timeout key to `locals.tf` `definitions.resource_timeouts` if the resource supports that action.

### Tags
Use `var.tags` directly (merged with module defaults at the calling layer, not inside the module):
```hcl
tags = var.tags
```

### Variables
- Use `optional()` with sensible defaults for all non-required nested object fields
- Required variables (no default): `name`, `resource_group_name`, `location`, `administrator_login`, `administrator_password`
- Group related block arguments into typed `object()` variables (e.g., `backup`, `high_availability`, `maintenance_window`)
- **Never use `version` as a variable name** — it is a reserved word in Terraform. Use `postgresql_version` instead
- `administrator_password` must always be `sensitive = true`

### Outputs
Return the full resource object so callers can access any attribute. Mark sensitive if the resource embeds credentials:
```hcl
output "postgresql_flexible_server" {
  description = "The full azurerm_postgresql_flexible_server resource object."
  sensitive   = true
  value       = azurerm_postgresql_flexible_server.this
}
```

### metadata.tf
- **Never modify** — it defines `local.metadata` merging definitions, var overrides, and defaults for `resource_timeouts`, `tags`, `validator_error_messages`, and `validator_expressions`
- Reference timeouts via `local.metadata.resource_timeouts[...]`

### azurerm v4.x breaking changes
In azurerm >= 4.0, some nested blocks were **flattened to top-level attributes**:
- `storage {}` block removed — use `storage_mb`, `auto_grow_enabled`, `storage_tier` as top-level attributes
- `network {}` block removed — use `delegated_subnet_id`, `private_dns_zone_id`, `public_network_access_enabled` as top-level attributes
- IDE language server (LSP) may show false "unexpected attribute" errors after provider upgrades; `terraform validate` is the authoritative check

## Security Considerations

These defaults are intentionally permissive for dev/test ease. Tighten for production:

| Variable | Default | Production recommendation |
|----------|---------|--------------------------|
| `public_network_access_enabled` | `true` | `false` + set `var.network` with VNet delegation |
| `authentication.active_directory_auth_enabled` | `false` | `true` + set `tenant_id` |
| `authentication.password_auth_enabled` | `true` | `false` once all clients use AAD tokens |
| `backup.geo_redundant_enabled` | `false` | `true` |
| `backup.retention_days` | `7` | `14`–`35` depending on RPO/compliance |
| `sku_name` | `B_Standard_B1ms` | `GP_Standard_D*` or `MO_Standard_E*` |

CMK encryption available via `var.customer_managed_key` + `var.identity` (requires `UserAssigned` identity).
